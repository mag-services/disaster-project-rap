ConfidenceSnapshot — New Model (DRMIS)

class ConfidenceSnapshot(models.Model):
    """Daily snapshot of field-check coverage per hazard event.
    Enables time-series charts showing confidence improving over deployment."""
    event          = models.ForeignKey(CycloneEvent, on_delete=models.CASCADE)
    snapshot_date  = models.DateField(auto_now_add=True, db_index=True)
    total_items    = models.PositiveIntegerField()
    model_count    = models.PositiveIntegerField()
    verified_count = models.PositiveIntegerField()
    adjusted_count = models.PositiveIntegerField()
    rejected_count = models.PositiveIntegerField()
    coverage_pct   = models.FloatField()   # (v+a+r)/total
    confidence_pct = models.FloatField()   # weighted score

    class Meta:
        unique_together = ["event", "snapshot_date"]
        ordering = ["-snapshot_date"]

⚙️ How RAP Improves Over Time

Field checks close the feedback loop between model-estimated damage and ground truth. Over multiple events, systematic bias per sector or intensity zone can be corrected in the multiplier inputs.

1

RAP runs → TabularItems created

Quarto RAP model uses intensity × baseline × damage multiplier to estimate physical/financial damage per council × sector. These become TabularItem records in DRMIS with confidence = model.

2

Field teams deploy → mobile observations synced

Assessors use the Android app to record roof/wall damage per asset. On sync, the app calls POST /api/v1/field-check/records/mobile/. The server matches observations to the relevant TabularItem by council × sector and creates FieldCheckRecord entries.

3

DRMIS shows discrepancies

The DRMIS coverage dashboard shows which items are adjusted and the gap between RAP estimate and observed value. For example: RAP estimated 30% education damage in Port Vila, but field checks averaged 58%. This signals the education damage multiplier for Category 4 is too low.

4

VBoS analyst reviews and updates 2a Input- Damage Multiplier.csv

Using the DRMIS adjustment data, the analyst updates the damage multiplier input file in the RAP repository. Specifically: observed_value from FieldCheckRecords (average over a council) ÷ intensity exposure factor = revised multiplier. A Multiplier Calibration Report can be generated automatically from the DRMIS API.

5

Next RAP run uses improved multipliers

The updated 2a Input- Damage Multiplier.csv feeds the next event's RAP run. Over time, as more events accumulate field check data, the multipliers converge toward real-world damage ratios. The DRMIS ConfidenceSnapshot time-series shows the coverage trajectory per event.

🧮 Multiplier Calibration Formula

Current RAP Formula

Estimated Damage =
  baseline_asset_count
  × intensity_multiplier[category]
  × damage_multiplier[sector]
  × replacement_cost_per_unit

damage_multiplier is the parameter field checks help calibrate. It represents the fraction of assets that sustain damage at a given intensity.

Field-Derived Correction

# For each council × sector with ≥ 3 adjusted checks:
field_avg_damage = mean(observed_value
                       for FieldCheckRecord
                       where status='adjusted')

model_damage_pct = TabularItem.value /
                   (baseline × replacement_cost)

correction_factor = field_avg_damage / model_damage_pct

new_multiplier = current_multiplier × correction_factor
⚠️ Minimum sample size recommended: 3+ field checks per council × sector per intensity band

Corrections based on fewer observations may reflect outliers rather than systematic bias. The DRMIS calibration report should flag low-sample adjustments.
